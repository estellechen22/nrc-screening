## 一、方案背景

基于 [creator_screening_prd.md](../creator_screening_prd.md) 中定义的六层筛查体系和已有的 FastAPI 后端（`/主播风险筛查/`），现规划一个**面向发行负责人日常使用**的 AI 背调筛查 Web 页面。核心目标：

- **输入**：粘贴创作者 YouTube/Twitch 等平台链接（支持批量换行输入）
- **处理**：调用后端 AI 筛查引擎，执行六层分析
- **输出**：不仅展示综合评分，更展示**每层详细评分、风险信号、关键词命中情况**，保留人工二次审查空间

## 三、AI 筛查流程详解（面向网页展示的核心逻辑）

### 3.1 第一步：URL 提交与平台识别

```
用户输入: https://www.youtube.com/@PokemonChallenges
          https://www.twitch.tv/smallant
          
系统处理:
  1. 按换行符拆分URL列表
  2. 逐个检测平台归属：
     - youtube.com / youtu.be → YouTube
     - twitch.tv → Twitch
     - twitter.com / x.com → Twitter/X
     - tiktok.com → TikTok
  3. 提取频道/用户唯一标识
  4. 不合法URL → 前端即时标红提示
```

## 四、创作者风险评估决策矩阵

### 4.1 最终判定标准

| 综合评分 | 判定 | 含义 | 操作建议 |
|:-------:|:----:|------|---------|

### 5.2 交互流程

```
【单输入模式】
用户粘贴1个URL → 点击"开始筛查" → 进入进度展示 → 完成后自动展开结果

【批量输入模式】
用户粘贴多个URL → 自动识别为N个 → 点击"开始筛查·共N人"
  → 进入批量进度页（显示 N/N completed）
  → 完成后展示汇总对比表
  → 点击单行 → 跳转到该创作者的详细结果（与单输入结果页复用）

【结果展示页交互】
- 顶部 sticky 综合评分卡片（始终可见）
- 下方六层卡片，**每层标题行始终展示评分 + 风险等级圆点（🟢🟡🔴）**
- **风险等级为 medium/high 的层默认自动展开**，low 的层默认折叠
- 点击层标题 → 展开/折叠该层全部细节（展开后的三层结构：子项评分明细 → 风险信号+证据链接 → 关键词标签云）
- 支持"全部展开/全部收起"一键切换
```

### 5.3 视觉风格

- **整体风格**：暗色主题（与现有 `creator_screening_dashboard.html` 一致）
- **颜色体系**：
  - 安全/通过：绿色 ✅
  - 警告/需关注：黄色 ⚠️
  - 危险/否决：红色 🚫
  - 品牌色：金色渐变（与洛克王国品牌一致）
- **评分环形图**：角度 + 颜色渐变表示分数高低
- **排版**：信息密集但结构清晰，卡片化设计

## 七、技术方案

| 项目 | 方案 | 理由 |
|------|------|------|
| 前端框架 | **原生 HTML/CSS/JS**（单页应用） | 无需构建工具链，直接本地打开或通过FastAPI静态文件服务 |

| 部署 | FastAPI static files serve 或 `npx serve` | 本地运行 |

## 八、已确认决策（2026-06-09）

### Q1: 批量输入的处理模式 → ✅ **方案A，限制最大并行5条**

采用并行提价方式，前端一次性提交 N 个任务，但同时活跃的后端任务最多 5 条，超出部分在前端排队等待。这样可以兼顾效率和 API 配额。

### Q2: 结果展示的默认展开深度 → ✅ **方式B：仅展开风险层，但各层评分始终可见**

- 各层卡片的标题行**始终展示该层的评分和风险等级圆点**（🟢🟡🔴），即使用户不展开也能一目了然
- 点击层标题可展开/折叠该层详情
- 风险等级为 medium/high 的层**默认自动展开**
- 风险等级为 low 的层**默认折叠**，用户可手动点击展开

### Q3: 平台支持范围 → ✅ **仅 YouTube，但保留架构扩展性**

当前仅支持 YouTube。但前端架构按**平台无关**设计：
- URL 输入框不做平台限制
- 每个 URL 提交时附带 `platform=youtube` 参数
- 后续如果要加 Twitch/Twitter/TikTok，只需：
  1. 前端：更新 `detect_platform()` 函数，新增平台识别规则
  2. 后端：新增对应平台的 L1/L2 数据采集服务
  3. 数据库：已有 `platform` 字段，无需改表

可行性：**完全可行，且改动成本低**。

### Q4: 导出格式 → ✅ **Excel 中包含简报摘要**

不需要 PDF，但 Excel 导出包含两个 Sheet：

| Sheet | 内容 |
|-------|------|
| **Sheet 1 摘要** | 每条记录一行：创作者名、平台、评分、判定、一票否决、Top 3 风险点（一句话简报） |
| **Sheet 2 详情** | 六层的详细评分、信号、关键词（与网页展示一致） |

### Q5: 人工复核流程 → ❌ **不需要**
